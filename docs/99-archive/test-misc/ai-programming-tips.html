一、AI 编程思维重塑

AI 编程不是"让 AI 帮我写代码"，而是"让 AI 帮我思考，我来做决定"。这是两个完全不同的模式。

1.1 核心认知转变

传统编程 AI 辅助编程
我是程序员，AI 是工具 我是架构师，AI 是高级初级工程师
一行一行写代码 设计需求 → AI 生成 → 我审查 → 迭代优化
遇到问题查 Stack Overflow 直接把代码丢给 AI，让它分析
自己测试 让 AI 写测试用例，我来验证
一个人搞定一切 多人分工：架构师 + 代码审查员 + 测试员
⚠️ 最重要的心态转变：
"AI 帮我节省了 80% 的时间，剩下的 20% 我来打磨。"
不要指望 AI 一次出完美代码——迭代才是核心能力。

1.2 正确的角色定位

在 AI 辅助编程中，你需要扮演 三个角色：

  1. 产品经理（PM） —— 明确需求、定义边界、验收交付物
  2. 架构师（Architect） —— 设计技术选型、模块划分、数据流向
  3. 代码审查员（Reviewer） —— 逐行检查 AI 生成的代码，确保正确性和安全性

AI 是你的高级工程师，它干活，你把关。这个分工不能反。

二、提示词工程核心技巧

提示词（Prompt）质量直接决定 AI 输出质量。以下是 2026 年经过社区验证的最佳实践。

2.1 万能提示词模板

[角色]
你是一位资深 {语言} 开发工程师，精通 {框架/库}。

[背景]
我正在开发一个 {项目描述}，使用 {技术栈}。

[需求]
请实现 {具体功能}，要求：
1. {验收标准1}
2. {验收标准2}
3. {验收标准3}

[约束]
- 不要使用 {不想要的内容}
- 必须遵循 {项目规范}
- 代码需要包含 {注释/类型/文档}

[示例]
参考以下代码风格：
{粘贴示例代码}

[输出]
请输出完整代码，并在代码后附上：
1. 使用说明
2. 已知限制

2.2 六大核心技巧

技巧 说明 示例
指定角色 给 AI 一个专业身份 "你是一位安全专家..."
提供上下文 给足够的项目背景 "这是一个 Flutter 项目，使用 Riverpod..."
分步思考 要求 AI 一步步分析 "先分析需求，再设计架构，最后写代码"
给出示例 用示例代码定义期望 "参考以下代码风格：..."
明确约束 告诉 AI 不要做什么 "不要使用任何 any 类型"
要求输出格式 规定输出结构 "输出完整代码 + 使用说明 + 已知限制"

2.3 错误提示词 vs 正确提示词

❌ 错误示例：
"帮我写一个登录功能"
✅ 正确示例：
"我需要一个 Flutter 项目的用户登录功能，使用 Firebase Auth。
要求：
1. 支持邮箱 + 密码登录
2. 有加载状态和错误提示
3. 使用 Riverpod 做状态管理
4. 输入框有基本验证（邮箱格式、密码≥6位）
5. 参考现有项目的代码风格（已附示例）
请先列出实现步骤，确认后再输出代码。"

2.4 复杂任务拆解技巧

遇到复杂功能时，不要一次丢一个大需求，拆成多个小步骤：

第一步：生成数据模型（User, Order, Item）
第二步：生成 API 接口层
第三步：生成 UI 组件
第四步：集成测试

每步确认后再进入下一步。这样 AI 的输出质量更高，你也更容易审查。

三、开发环境配置最佳实践

3.1 项目级配置（Rules / .cursorrules）

在根目录创建 .cursorrules 或 .cursor/rules.md，定义项目规范：

# 项目规范
- 语言：Dart 3.5+
- 框架：Flutter 3.24
- 状态管理：Riverpod
- 路由：go_router
- 命名：文件使用 snake_case，类使用 PascalCase
- 错误：统一使用 Result 模式，返回 {success, data, error}
- 禁止：使用 any 类型、魔法字符串、硬编码 URL
- 必须：每个函数有 doc comment、每个 widget 有注释说明用途

这样每次对话时，AI 自动遵守项目规范，不需要重复说明。

3.2 AI 模型配置策略

根据任务类型切换不同的 AI 模型：

任务类型 推荐模型 原因
日常代码补全 轻量模型 / Tab 自动补全 速度快，成本低
复杂架构设计 最强模型（Claude / GPT-4o） 需要深度推理
代码审查 推理能力强的模型 需要发现隐藏问题
调试错误 带代码分析能力的模型 需要理解错误上下文
文档生成 文本生成能力强的模型 需要清晰表达
💡 省钱技巧：日常用本地模型（免费），重要任务切云端最强模型。混合使用，效果和成本兼顾。

3.3 Git 集成最佳实践

# 每次 AI 生成代码后，养成习惯：
1. 先审查代码（逐行看）
2. 本地运行测试
3. git add + commit 后再进入下一步
4. commit message 注明 AI 参与了哪些修改

这样出了问题可以 git bisect 快速定位，也方便回滚。

四、项目级 AI 协作工作流

4.1 从需求到上线的完整流程

📋 标准流程：
需求分析 → 架构设计 → 代码生成 → 审查 → 测试 → 迭代 → 上线

阶段 1：需求分析

提示词：
"我需要一个{项目描述}，目标用户是{用户群体}，
核心功能是{功能1、2、3}。请帮我：
1. 列出用户故事（User Stories）
2. 列出非功能需求（性能、安全等）
3. 列出 MVP 范围（最小可发布版本）
4. 列出后续迭代方向"

阶段 2：架构设计

提示词：
"基于以下需求：{粘贴需求分析}，请帮我设计：
1. 技术选型及理由
2. 模块划分（包含依赖关系图）
3. 数据模型设计
4. API 接口设计
5. 目录结构
请先给出方案，我确认后再细化。"

阶段 3：代码生成（按模块逐个来）

提示词：
"请先实现数据模型层。
技术栈：{列出}
参考项目现有代码风格：{粘贴示例}
输出：完整代码 + 说明"

阶段 4：审查与测试

提示词：
"请作为代码审查员，审查以下代码：
1. 是否存在安全风险？
2. 是否有性能问题？
3. 是否符合项目规范？
4. 是否有更好的写法？
5. 请生成对应的单元测试"

4.2 多 Agent 协作模式

对于大型项目，可以同时启动多个 AI 对话，分工协作：

角色 负责内容 对话方式
架构 Agent 整体设计、技术方案 一次对话定方案
前端 Agent UI 组件、状态管理 按组件逐个生成
后端 Agent API、数据库、业务逻辑 按接口逐个生成
测试 Agent 单元测试、集成测试 拿到代码后自动测试
💡 实际场景：你在一个对话里让 AI 设计架构，在另一个对话里让同一个 AI 生成 UI 代码，在第三个对话里让它审查后端代码。三个对话独立运行，互不干扰。

五、代码质量与安全

5.1 AI 生成代码的六大必查项

# 检查项 说明 严重程度
1 API 密钥泄露 硬编码的密钥、Token 必须抽离到环境变量 🔴 高
2 SQL 注入 拼接 SQL 字符串而非使用参数化查询 🔴 高
3 XSS 漏洞 直接渲染用户输入未转义 🔴 高
4 异常处理缺失 try-catch 不完整，静默吞掉异常 🟡 中
5 类型安全问题 使用 any、dynamic 导致运行时崩溃 🟡 中
6 性能隐患 N+1 查询、未缓存的重复计算 🟢 低

5.2 安全最佳实践

🚫 绝对不要做：
- 把 AI 生成代码中的 API 密钥直接提交到 Git
- 把生产环境数据库密码告诉 AI
- 让 AI 访问你的真实用户数据
✅ 应该这样做：
- 使用测试数据（伪造数据）让 AI 生成代码
- 密钥全部走环境变量 / 密钥管理服务
- 用 .env.example 文件展示密钥格式，不填真实值

5.3 让 AI 做代码审查

每次 AI 生成代码后，再给它一段审查提示词：

"请审查以下代码，重点关注：
1. 安全风险（注入、XSS、CSRF 等）
2. 边界条件处理（空值、越界、超时）
3. 性能问题（循环内查询、不必要的计算）
4. 代码风格一致性
5. 是否有更优实现方案
请逐条列出问题和建议。"

六、常见陷阱与避坑指南

6.1 新手最常犯的 8 个错误

错误 后果 正确做法
一次给太多需求 AI 输出混乱，部分功能被忽略 拆成小步骤，逐个完成
不审查直接运行 Bug 多，调试成本高 逐行审查 + 单元测试
不更新上下文 AI 用旧代码生成新代码 每次对话都告知当前项目状态
过度依赖一个工具 工具限制 = 你的上限 多工具交叉验证
不写测试 改了一处，坏了三处 AI 写测试，你验证测试
不学基础 无法审查 AI 代码 基础能力是审查能力的底气
复制粘贴不读 引入无用代码、安全风险 每行代码都要理解
不保留版本 改乱了无法回退 频繁 commit，善用 Git

6.2 "AI 幻觉" 应对策略

AI 有时会"一本正经地胡说八道"——引用不存在的 API、编造不存在的库。

应对方法：
1. 对 AI 输出的 API/库/版本做验证（查官方文档）
2. 让 AI 给出代码后，用 `dart pub deps` / `npm list` 验证
3. 如果编译报错，把错误信息丢给 AI 让它修复
4. 重要功能先用最小 demo 验证思路，再集成
⚠️ 记住：AI 说"这个 API 存在于 Flutter 3.24"——你去 dart.dev 查一下再相信。

七、进阶：Agent 与 MCP

7.1 MCP（模型上下文协议）

MCP 让 AI 能访问外部工具和数据源。相当于给 AI 装上"手和脚"：

能力 说明 示例 MCP
📁 文件系统 读写项目文件 filesystem MCP
🗄️ 数据库 查询/操作数据库 postgres MCP
🔧 终端 执行命令行命令 shell MCP
🌐 网络 调用 API / 爬取数据 web-fetch MCP
📊 代码分析 静态分析 / 重构 eslint MCP

7.2 Skills（技能定义）

Skills 是用 Markdown 文件定义的"专家模式"：

# skills/database/SKILL.md
# 数据库专家
当你处理数据库相关问题时，遵循以下规范：
1. 使用参数化查询防止 SQL 注入
2. 每个查询必须有超时限制（30s）
3. 数据库迁移使用 migration 文件管理
4. 索引设计原则：...
5. 连接池大小 = CPU 核心数 × 2

配置后，当你讨论数据库问题时，AI 会自动应用这些规则。

7.3 Hooks（自动化钩子）

让 AI 在特定操作时自动执行任务：

{
  "onSave": [
    "npx prettier --write {file}",
    "npx eslint --fix {file}"
  ],
  "beforeCommit": [
    "npm test",
    "git diff --stat"
  ]
}

九、AI 编程 Checklist

每次使用 AI 辅助编程时，按这个清单逐项检查：

🔵 写提示词之前

    □ 我是否已经理解了需求的完整范围？
    □ 我是否准备好了上下文（技术栈、代码示例、项目规范）？
    □ 这个需求是否可以拆成更小的步骤？

🟡 收到 AI 输出后

    □ 我逐行审查了代码吗？
    □ API 调用是否正确？版本是否匹配？
    □ 有没有安全漏洞（密钥泄露、注入）？
    □ 有没有多余的、不需要的代码？
    □ 错误处理是否完整？
    □ 代码风格是否符合项目规范？

🟢 运行和测试后

    □ 本地编译/运行成功了吗？
    □ 单元测试通过了吗？
    □ Edge case（边界条件）测试了吗？
    □ git commit 了吗？
    □ 更新了项目文档吗？
📌 最后的话

AI 编程工具再强大，也改变不了一个事实：好软件需要时间打磨。

依然需要你亲自动手。

把 AI 当助手，不是替代品。慢慢来，享受过程。
