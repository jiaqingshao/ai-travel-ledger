# ADR-007: R012 解决方案选型 — 切换到 腾讯云开发 (CloudBase)

**状态**: ❌ **已撤销**（被 [ADR-008](ADR-008-phase1-local-only-cloud-deferred.md) 取代）
**日期**: 2026-07-15 (决策) → 2026-07-15 (撤销)
**决策者**: 创始人 + 主 Agent
**优先级**: 高（V1.3 国内 Android 上架前置条件）
**关联**: [ADR-005 国内 Android only](ADR-005-android-cn-only.md), [R012 风险登记](../../03-management/risk-register.md)

---

## 背景

V1.2 cloud-milestone (2026-07-14) 使用 **Supabase 海外**作为云同步后端。

**R012（合规风险）**：
- 数据存储在境外（Supabase US/EU region）
- V1.3 国内 Android 上架后，国内用户数据合规边界需评估
- 《个人信息保护法》《数据安全法》《网络安全法》要求境内化运营
- 国内 Android 应用商店上架**可能要求数据境内存储**（华为/小米严格）

**用户决策约束（2026-07-15）**：
- 必须 **国内云服务商**
- 必须 **免费额度**（V1.3-V2.0 长期 0 月费）

## 决策

**⚠️ 此决策仅 18 分钟有效 (2026-07-15 11:25 → 13:50)**

**采纳方案 A：腾讯云 CloudBase (TCB)**

**撤销原因** （2026-07-15 13:50）:
- 创始人复查: TCB 个人版实际为 **¥19.9/月**, 不是免费
- 微信云开发（真正免费）6-8 周迁移工作量大
- 改采 ADR-008: Phase 1 纯本地模式, 云功能 V2.0 启用

本文档保留作为决策历史 (TCB vs 华为 vs 阿里 vs 自建对比分析 + 5-7 周迁移工作量估算 都有参考价值)
详细撤销原因与新决策: [ADR-008](ADR-008-phase1-local-only-cloud-deferred.md)

### 决策依据

1. **唯一同时满足"国内 + 免费 + 够用"三约束**
   - 数据库 2 GB 是华为 50 MB 的 40 倍
   - 个人开发者实名即可使用（不用企业认证）
2. **Flutter SDK 成熟**（`cloudbase_sdk` 官方包）
3. **架构最接近当前 Supabase**（迁移工作量最小）
4. **国内 BaaS 头部**（微信生态 + 实名后 0 月费）

### 接受的变化

| 失去 | 替代 |
|---|---|
| 全球分发（海外市场） | 国内合规 + ¥0 月费 |
| 多账号灵活性 | 个人实名绑定（接受） |
| Supabase 5 分钟文档查询 | TCB 文档略弱（接受） |

### 不影响

- 当前 V1.2 milestone 已发布的 APK（仍用旧 Supabase，可继续用）
- `lib/core/supabase/` 等代码（迁移过程中并行）
- 全部 lib/ 测试（迁移到 TCB 后逐个验证）

## 关键技术决策（待细节设计时决）

### K-1: 数据库选 TCB PostgreSQL 还是 MongoDB

- **PG**：与当前 Supabase schema 兼容，迁移脚本简单（保持 7 张表结构）
- **MongoDB**：更灵活，但 schema 重写
- **MySQL**：折中
- **推荐**：**PG**（迁移成本最小，与 PostgreSQL 知识复用）

### K-2: 认证方式

- 邮箱 + 密码（与 Supabase Auth 一致）
- 微信扫码（增加依赖）
- 匿名（不需要注册，临时账号）
- **推荐**：**邮箱 + 密码**为主，留接口给未来微信扫码

### K-3: RLS 翻译

- Supabase RLS 用 SQL 表达式 + auth.uid()
- TCB 安全规则是独立 DSL（不完全等价）
- 需要重新写 7 张表的访问控制策略
- **策略**：先实现"创始人账号 = 无 RLS"（V1.3 单用户阶段），V2.0 多人时再细化
