## 风险详情

### R001 - 创始人编程经验不足 ⚠️ 极高

**描述**:
- 创始人有 C、Python 基础
- 无 Flutter / Dart / 移动开发经验
- 无后端 / 数据库 / DevOps 经验

**影响**:
- 进度可能严重延迟
- 代码质量可能不高
- 架构决策可能走弯路

**应对措施**:
1. **AI 工具最大化** - Trae AI Builder 模式 + Composer Agent
2. **本地模型** - Qwen3.6 35B 几乎 0 token 费
3. **外部支持** - 需要时雇 Flutter 兼职做 Code Review
4. **分阶段** - 先 MVP，不追求完美架构
5. **记录决策** - 所有技术选型写 ADR，未来不返工

**状态**: 监控中，持续到 MVP 发布

### R002 - 项目周期延长 ⚠️ 高

**描述**:
- 学习 + 开发双线作战
- 容易陷入"完美主义"

**应对措施**:
- 严格 MVP 范围，P0 优先
- 2 周一次复盘
- 砍需求原则：用户说"想要"≠ "必须做"

**关键里程碑**:
- T+2 月 MVP 内测
- T+3 月 1.0 上架
- 超期 > 1 月：触发需求砍剪

### R004 - Supabase 数据在境外 (合规风险升高级) ⚠️ 高

**描述**:
- 当前 Supabase 项目位于境外（海外 BaaS）
- V1.2 cloud-milestone 用户数据已托管在境外
- 国内 Android 上架后，国内用户数据合规边界需评估
- 《个人信息保护法》《数据安全法》《网络安全法》要求

**影响**:
- 华为/小米/部分商店可能要求数据境内存储
- V1.3 上架时可能受阻
- 即使上架成功，存在被举报/下架风险

**应对措施**:
1. **V1.3 启动前**评估国内云替代（阿里云 / 腾讯云 Supabase 自建）
2. **过渡方案**：海外 Supabase + 用户告知（接受度低时再切）
3. **如果切国内**：自建 Supabase（Docker + PostgreSQL），或者用腾讯云开发
4. **决策日期**: ADR-005 风险升级（2026-07-15）

**状态**: 待评估（**V1.3 启动前必须决**）

### R011 - 软件著作权 (国内主流商店上架需) 🆕 ⚠️ 高

**描述**:
- 国内主流 Android 商店（华为/小米/OPPO/vivo/应用宝）上架审核要求**软件著作权**（软著）
- 软著申请周期: 1-2 个月（普通件），加急 3-5 天（约 ¥800 额外）
- 软著申请费用: ¥300-500（基础件） + 可能 ¥300-600 加急
- 个人开发者申请软著**可以通过**（需提供身份证 + 源代码）

**影响**:
- V1.3 国内上架**前置条件**
- 1-2 月等待期（在申请期间无法上架）
- 增加预算（¥300-800）

**应对措施**:
1. **V1.3 启动第 1 周** 即开始申请软著（备齐源代码 + 文档）
2. 选择价格合适的代理（淘宝 ¥300 / 国家版权局 ¥500 / 第三方加速 ¥800）
3. **不阻塞** V1.3 代码开发（与开发并行）

**状态**: 待启动（V1.3 启动前必须完）

**决策记录**: [ADR-005](02-architecture/04-adr/ADR-005-android-cn-only.md)

### R012 - ICP 备案 (云端记账功能合规) 🆕 ⚠️ 高 → ✅ 决策完成 (ADR-007, 2026-07-15)

**描述**:
- 任何提供**网络服务**的 App 在中国大陆运营需 ICP 备案
- 当前 AI 旅行账本使用 Supabase 境外云（未做 ICP）
- 国内 Android 商店上架审核：*部分商店*要求检查 ICP 备案
- 《网络安全法》明确规定：未备案擅自提供互联网服务可处罚款甚至停业整顿

**影响**:
- V1.3 国内上架**阻塞风险**
- 即使上架成功，存在被举报/下架风险

**决策结果**: 选 **方案 A 演化版** —— **腾讯云开发 (CloudBase)**

1. **国内云服务商**: 腾讯云开发（CloudBase），数据在腾讯云上海/广州机房
2. **免费层**: 数据库 2 GB + 存储 5 GB + 云函数 4 万 GBs/月（V1.3-V2.0 够用）
3. **个人开发者友好**: 实名（身份证）即可用，不需企业认证
4. **仍需 ICP 备案**: 但变成备案腾讯云提供的子域名 `xxx.tcloudbaseapp.com`，**腾讯云为你代备案**（免费）

**新决策结果**:
- **R012 状态**: ✅ **决策完成** (ADR-007, 2026-07-15)
- **执行待办**:
  1. 创始人注册腾讯云 + 实名 (Step 1-2, 见 [tcb-setup-guide.md](tcb-setup-guide.md))
  2. 创始人创建 CloudBase 环境 (Step 3)
  3. 创始人提供密钥给主 Agent (Step 4-6)
  4. 主 Agent 启动 5-7 周迁移 (SQL + Auth + Storage + Realtime)
  5. ICP 备案 V1.3 上架前办（1-2 周，腾讯云代办）

**决策详情**: [ADR-007](../02-architecture/04-adr/ADR-007-r012-tcb-migration.md)

**申请流程**: [tcb-setup-guide.md](tcb-setup-guide.md) (面向创始人，含 URL + 步骤 + 命令)

### R014 - TCB 迁移风险 (ADR-007) 🆕 ⚠️ 高

**描述**:
- 决策后实际迁移面临的多重风险:

| 子风险 | 概率 | 影响 | 等级 |
|---|---|---|---|
| **5-7 周工期超期** | 中 | 高 | 高 |
| **数据迁移期间数据丢失** | 低 | 高 | 中 |
| **TCB 实名后换账号难** | 高 | 中 | 中 |
| **海外市场无法使用** | 高 | 低 | 低 |
| **TCB 文档/SDK 不足** | 中 | 中 | 中 |
| **PG 版本差异（Supabase 15.x vs TCB PG 版本）** | 中 | 中 | 中 |

**影响**:
- V1.3 时间表被压缩（原 4-6 周 → 5-7 周）
- 迁移期（7-19 至 9-05）期间，用户数据需要双写（Supabase + TCB）防止丢失

**应对措施**:
1. **双写期**: 迁移期间 Dart 端同时写两个后端，确保零数据丢失
2. **分阶段切换**: 先 dev 环境验证 → prod 环境试运行 → 全量切换
3. **回退预案**: 保留 Supabase 配置，TCB 出问题时秒切回去
4. **备份策略**: TCB 数据每日自动备份 + 本地 archive
5. **进度跟踪**: 创建 ISSUE-036 跟踪迁移任务
6. **联系人**: TCB 有工单系统 / 微信群支持，重大问题可快速得到回复

**状态**: 监控中（迁移启动后每 3 天 review）

**决策记录**: [ADR-007](../02-architecture/04-adr/ADR-007-r012-tcb-migration.md)

### R013 - 多商店审核差异 🆕 ⚠️ 中

**描述**:
- 每个国内 Android 商店审核标准、政策、IAP 抽成不同
- 主流商店 IAP 抽成参考：小应用市场 5%，主流商店 30-50%
- 适配多商店需要：单独审核包 + 截图 + 文案 + 隐私政策
- 每个商店上架审核周期：3-15 工作日

**影响**:
- V1.3 上架工作量比预期大
- 高级版收费需考虑各商店 IAP 抽成（¥18/月 → 实收约 ¥9-12）

**应对措施**:
1. V1.3 优先选 1-2 个主力商店试水（如小米 + OPPO，对个人友好）
2. 各商店 IAP 单独接入（小应用市场直接支付宝/微信；主流商店用商店支付渠道）
3. 隐私政策 + 用户协议文案统一，各商店单独提交
4. V1.4 再扩展到 5+ 商店（覆盖 80% 国内 Android）

**状态**: 待评估（V1.3 设计阶段）

**决策记录**: [ADR-005](02-architecture/04-adr/ADR-005-android-cn-only.md)
