## 一、团队架构与职责

### 1.1 组织架构

```
┌─────────────┐
│    用户      │  （产品负责人 / 决策者）
│   创始人     │
└──────┬──────┘
       │ 需求确认 / 阶段验收
       ▼
┌─────────────┐
│   PM（主     │  需求拆解 / 任务分发 / 进度跟踪 / 代码审查 / 质量把关
│   Agent）   │
└──┬─────┘
   │
   ▼
┌─────────────┐
│  开发（全栈  │  前端 + 后端全栈开发 / UI 页面 / 数据模型 / 组件 / 路由 / 动画 / 交互逻辑
│   Agent）   │  （合并原开发 A + 开发 B，统一上下文）
└──┬─────┘
   │
   ▼
┌─────────────┐
│  测试（QA   │  自动化测试 + 人工抽检 / Bug 报告 / 质量门禁
│   Agent）   │
└─────────────┘
```

### 1.2 角色职责

| 角色 | 代号 | 核心职责 | 产出物 |
|---|---|---|---|
| **PM（主 Agent）** | 指挥官 | 需求拆解、任务分发、进度跟踪、代码审查、质量验收、向您汇报 | 开发日报、阶段报告、Bug 汇总 |
| **开发（全栈）** | 全栈开发 | 前端+后端全栈开发（数据模型、UI 页面、组件、路由、动画、交互逻辑、结算引擎、Hive/Supabase 接入） | 全栈代码、单元测试、UI 页面、组件库 |
| **测试（QA）** | QA 专员 | 自动化测试、人工抽检、对照 PRD/FSD 找 Bug、提 Bug 报告 | 测试报告、Bug 清单、质量门禁 |

**合并理由**：
- 减少沟通成本，开发 Agent 统一上下文理解
- 前后端职责合并，避免交接丢失
- 更适合小型项目快速迭代

### 1.3 协作流程

```
1. 您确认需求 → PM 拆解 Task
2. PM 分配任务 → 开发（全栈）统一开发
3. 开发完成 → 提交代码（git commit）
4. QA 入场 → 测试 + 审查 → 输出 Bug 报告
5. PM 审核 Bug 报告 → 分配修复任务给开发
6. 修复完成 → QA 复测 → 签字放行
7. PM 合并代码 → 推送到 GitHub → 向您汇报进度
```

## 二、代码管理规范

### 2.1 版本控制

| 项目 | 规范 |
|---|---|
| **工具** | Git + GitHub |
| **主分支** | `main`（永远保持可运行状态，只有 PM 有权限合并） |
| **开发分支** | `dev`（所有 Agent 在此分支工作） |
| **功能分支** | `feature/xxx`（每个新功能独立分支） |
| **备份频率** | 每个 Phase 完成后推送 GitHub 并打 Tag（如 `v0.1.0-alpha`） |

### 2.2 提交规范（Atomic Commits）

**原则：原子化、语义化、可回滚**

| 规则 | 说明 |
|---|---|
| **粒度** | 每个小功能完成即提交，禁止写完整个项目再提交 |
| **格式** | `type: description` |
| **Type 枚举** | `feat` / `fix` / `refactor` / `docs` / `test` / `chore` |

**示例**：
```
feat: 新增结算引擎贪心算法
fix: 修复重复检测逻辑中的空指针异常
refactor: 重构数据模型字段命名
docs: 更新 PRD 需求文档 v0.2
test: 添加结算引擎单元测试
chore: 配置 .gitignore 文件
```

### 2.3 分支保护规则

| 规则 | 说明 |
|---|---|
| **main 保护** | 只有 PM 可以合并到 main |
| **CI 检查** | 合并前必须通过 `flutter analyze` 和单元测试 |
| **QA 门禁** | 合并前必须有 QA 的测试通过报告 |
| **禁止直推** | 开发和测试 Agent 禁止直接推送到 main |

### 2.4 敏感信息管理

**`.gitignore` 必须包含**：
```
.env
*.key
*.pem
supabase/.env
```

**API Key 等敏感信息**：
- 禁止硬编码在代码中
- 通过环境变量或配置文件管理
- 配置文件加入 `.gitignore`

## 四、开发流程规范

### 4.1 Phase 划分

| Phase | 名称 | 预计时长 | 交付物 |
|---|---|---|---|
| **Phase 1** | 数据层 + 逻辑层 | 2 小时 | 数据模型、结算引擎、Hive、Supabase、单元测试 |
| **Phase 2** | UI 页面实现 | 3 小时 | 所有页面、组件、路由、动画、交互 |
| **Phase 3** | 云端同步 + 打磨 | 2 小时 | Supabase 同步、重复检测、错误处理、端到端测试 |

### 4.2 每个 Phase 的执行步骤

```
1. PM 拆解 Task 清单 → 您确认
2. PM 分配任务给 开发（全栈）
3. 开发统一开发（前后端一体）
4. 开发完成 → 提交代码（git commit）
5. PM 代码审查
6. QA 入场测试
7. 有 Bug → PM 分配修复 → 开发修复 → QA 复测 → 重复直到通过
8. 无 Bug → QA 签字放行
9. PM 合并代码 → 推送到 GitHub
10. PM 向您汇报 Phase 完成情况
```

### 4.3 代码审查（Code Review）

**审查内容**：
- [ ] 是否符合 PRD/FSD 需求
- [ ] 代码风格是否统一
- [ ] 是否有明显的逻辑错误
- [ ] 是否有安全漏洞
- [ ] 是否有性能问题
- [ ] 单元测试是否覆盖

**审查标准**：
- **P0 Bug**：功能不可用 → 必须修复才能合并
- **P1 Bug**：功能异常 → 必须修复才能合并
- **P2 Bug**：体验问题 → 建议修复，可延期
- **P3 Bug**：代码优化 → 建议修复，可延期

## 五、QA 测试规范

### 5.1 测试策略

| 测试类型 | 负责人 | 内容 |
|---|---|---|
| **单元测试** | 开发（全栈） | 数据模型、结算引擎、算法逻辑、组件测试 |
| **集成测试** | QA Agent | 跨模块功能、数据流 |
| **E2E 测试** | QA Agent | 端到端流程、用户体验 |
| **人工抽检** | QA Agent | 对照 PRD/FSD 逐项检查 |

### 5.2 Bug 管理

> **所有问题必须在 [`issue-tracker.md`](issue-tracker.md) 跟踪**（不含括杂的任务级故障，后者在 event-log.md）。issue-tracker 是问题跟踪的唯一权威来源。

**Bug 等级**：

| 等级 | 定义 | 处理时效 |
|---|---|---|
| **P0 - 致命** | 功能不可用、崩溃、数据丢失 | 立即修复 |
| **P1 - 严重** | 主要功能异常、影响核心流程 | 当前 Phase 内修复 |
| **P2 - 一般** | 次要功能异常、体验问题 | 可延期到下一 Phase |
| **P3 - 轻微** | 文案错误、样式微调 | 可延期到 V1.1 |

**Bug 报告模板**：
```markdown
## Bug 报告

- **Bug ID**: BUG-001
- **等级**: P1
- **模块**: 结算引擎
- **描述**: 贪心算法在 3 人场景下出现除零错误
- **复现步骤**:
  1. 创建旅程
  2. 添加 3 个成员
  3. 记录 1 笔账目
  4. 点击结算
- **预期结果**: 正常显示结算单
- **实际结果**: 崩溃，显示除零错误
- **日志**: [粘贴日志]
- **修复建议**: 检查空值判断
```

### 5.3 质量门禁

**合并到 main 的必须条件**：
- [ ] `flutter analyze` 0 errors
- [ ] 单元测试覆盖率 > 60%
- [ ] QA 测试通过报告
- [ ] PM 代码审查通过
- [ ] 无 P0/P1 Bug

## 六、Agent KPI 指标

### 6.1 PM（主 Agent）KPI

| 指标 | 定义 | 目标值 | 考核周期 |
|---|---|---|---|
| **需求拆解准确率** | Task 描述清晰、无歧义的比例 | ≥ 90% | 每个 Phase |
| **进度跟踪及时率** | 按时汇报进度的比例 | 100% | 每日 |
| **Bug 漏检率** | QA 发现的 Bug 中 PM 未审查到的比例 | ≤ 10% | 每个 Phase |
| **代码审查覆盖率** | PM 审查过的代码比例 | ≥ 80% | 每个 Phase |
| **文档更新及时率** | 需求变更后文档同步更新的比例 | 100% | 每次变更 |

### 6.2 开发（全栈）KPI

| 指标 | 定义 | 目标值 | 考核周期 |
|---|---|---|---|
| **代码质量** | `flutter analyze` 0 errors 的比例 | 100% | 每次提交 |
| **单元测试覆盖率** | 测试代码覆盖的行数比例 | ≥ 70% | 每个 Phase |
| **Bug 率** | 每千行代码的 Bug 数量 | ≤ 3 | 每个 Phase |
| **任务完成率** | 按时完成 Task 的比例 | ≥ 90% | 每个 Phase |
| **代码规范遵循率** | 符合提交规范、命名规范的比例 | ≥ 95% | 每次提交 |
| **UI 还原度** | 实际 UI 与 PRD/FSD 设计的一致比例 | ≥ 90% | 每个页面 |
| **组件复用率** | 可复用组件的比例 | ≥ 50% | 每个 Phase |
| **性能指标** | 页面加载时间、帧率等 | 启动 <2s、滚动 60fps | 每个 Phase |
| **暗色模式适配率** | 支持暗色模式的组件比例 | 100% | 每个 Phase |

### 6.3 测试（QA）KPI

| 指标 | 定义 | 目标值 | 考核周期 |
|---|---|---|---|
| **Bug 检出率** | QA 发现的 Bug 占总 Bug 的比例 | ≥ 80% | 每个 Phase |
| **测试覆盖率** | 测试覆盖的功能模块比例 | ≥ 80% | 每个 Phase |
| **漏测率** | 合并后用户发现的 Bug 占总 Bug 的比例 | ≤ 5% | 每个 Phase |
| **测试报告及时率** | 按时输出测试报告的比例 | 100% | 每个 Phase |
| **自动化测试比例** | 自动化测试用例占总用例的比例 | ≥ 60% | 每个 Phase |

## 九、变更管理
